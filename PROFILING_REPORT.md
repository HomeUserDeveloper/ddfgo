1. ОПТИМИЗАЦИЯ СКАНИРОВАНИЯ ДИРЕКТОРИИ:
   
   Проблема:
   - filepath.Walk создает много временных объектов
   - Бенчмарк показал 13,796 аллокаций на 1000 файлов
   - Это ~13.8 аллокаций на файл
   
   Решение:
   - Переиспользовать буферы при сканировании
   - Уменьшить кол-во создаваемых структур
   
   Код для оптимизации:
   ```go
   // Вместо создания slice для каждого уровня рекурсии,
   // использовать заранее выделенный буфер
   type dirScanner struct {
       pathBuffer   [1024]byte
       entryBuffer  [256]os.DirEntry
       fileInfoBuf  [256]os.FileInfo
   }
   ```
   
   Ожидаемый результат:
   - Сокращение аллокаций: 20-30%
   - Общее ускорение: 2-5%


2. ТОЧНОЕ ОПРЕДЕЛЕНИЕ БЫСТРОГО ХЕША:

   Проблема:
   - "5% начала + 5% конца" - эмпирическое значение
   - Для файлов 100-500 MB может быть недостаточно
   - Может привести к ложным положительным результатам
   
   Решение - Использовать адаптивный размер:
   ```go
   func calculateQuickHashSize(fileSize int64) int64 {
       const minBlockSize = 1024        // 1 KB
       const maxBlockSize = 10*1024*1024 // 10 MB
       
       // 0.5% для больших файлов > 1GB
       if fileSize > 1024*1024*1024 {
           blockSize := fileSize / 200
           if blockSize > maxBlockSize {
               blockSize = maxBlockSize
           }
           return blockSize
       }
       
       // 5% для малых файлов
       blockSize := fileSize / 20
       if blockSize < minBlockSize {
           blockSize = minBlockSize
       }
       if blockSize > maxBlockSize {
           blockSize = maxBlockSize
       }
       return blockSize
   }
   ```
   
   Ожидаемый результат:
   - Лучшая точность определения дубликатов
   - Общее улучшение надежности

3. ОПТИМИЗАЦИЯ ПАРАЛЛЕЛИЗМА:

   Проблема:
   - Текущее значение: runtime.NumCPU() * 2
   - На 16-ядерном процессоре = 32 горутины
   - Может быть неоптимально для I/O операций
   
   Решение - Адаптивное управление горутинами:
   ```go
   func getOptimalWorkerCount(totalFiles int64) int {
       cpuCount := runtime.NumCPU()
       
       // Для файлов < 1000: меньше горутин (I/O bound)
       if totalFiles < 1000 {
           return cpuCount
       }
       
       // Для файлов 1000-100K: оптимум
       if totalFiles < 100000 {
           return cpuCount * 2
       }
       
       // Для файлов > 100K: больше горутин (ограничение на I/O)
       return cpuCount * 4
   }
   ```
   
   Ожидаемый результат:
   - Лучшее использование ресурсов
   - Ускорение 5-15% в зависимости от размера данных


4. ОПТИМИЗАЦИЯ РАБОТЫ С ПАМЯТЬЮ:

   Проблема:
   - findDuplicatesByHash читает ВСЕ дубликаты в память
   - На 100K дубликатов может потребоваться большой буфер
   
   Решение - Потоковая обработка:
   ```go
   func findDuplicatesByHashStreaming(db *sql.DB) error {
       // Вместо:
       // 1. Выбрать все дубликаты
       // 2. Прочитать их в memory
       // 3. Обработать
       
       // Использовать:
       // 1. Выбрать дубликаты пакетами (batch = 1000)
       // 2. Обработать каждый пакет
       // 3. Перейти к следующему
   }
   ```
   
   Ожидаемый результат:
   - Масштабируемость на миллионы файлов
   - Константное использование памяти (не зависит от кол-ва дубликатов)
