ОПИСАНИЕ ПРИОРИТЕТА 3 ОПТИМИЗАЦИЙ

Три основных направления для следующей итерации оптимизации:

═══════════════════════════════════════════════════════════════════════════════

🔹 ОПЦИЯ 1: LRU КЕШ ДЛЯ ХЕШЕЙ ФАЙЛОВ

ПРОБЛЕМА:
  - При обработке больших наборов файлов некоторые файлы могут быть
    переобработаны (например, при повторных сканированиях)
  - Хеширование больших файлов занимает время

РЕШЕНИЕ:
  - LRU (Least Recently Used) кеш для хешей файлов
  - Хранить: path + mtime + size → quick_hash / full_hash

РЕАЛИЗАЦИЯ:
  Step 1: Двухуровневый кеш в памяти и в SQLite
    - In-memory LRU (1000-10000 записей для быстрого доступа)
    - Persistent кеш в SQLite для между-сессионного использования
  
  Step 2: Проверка перед хешированием
    IF file_exists && mtime+size matches cache:
      RETURN cached_hash
    ELSE:
      COMPUTE hash & UPDATE cache
  
  Step 3: Vacuum кеша при превышении лимита
    - Удалять старые файлы по LRU политике
    - Очищать кеш при --clean флаге

ОЖИДАЕТСЯ:
  📊 На повторных запусках: +5-10% производительность
  💾 Память: +2-5 MB для in-memory кеша
  ⏱️ Пример: 85 сек → 77-80 сек на повторных сканированиях

СЛОЖНОСТЬ:
  ⭐⭐⭐ Средняя (требует синхронизации, управления памятью)

════════════════════════════════════════════════════════════════════════════════

🔹 ОПЦИЯ 2: ПАРАЛЛЕЛЬНОЕ СКАНИРОВАНИЕ ДИРЕКТОРИЙ

ПРОБЛЕМА:
  - filepath.Walk() последовательно обходит директории
  - На системах с медленным диском это становится узким местом
  - I/O операции не параллелизируются между сканированием разных ветвей

РЕШЕНИЕ:
  - Разделить работу filepath.Walk на несколько параллельных горутин
  - Каждая горутина обходит отдельную ветку директории

РЕАЛИЗАЦИЯ:
  Step 1: Распределитель директорий
    func distributeDirectoriesParallel(rootDir string, workers int) []string
      - Список top-level директорий
      - Назначить каждой горутине свою поддиректорию
      - Использовать каналы для сбора результатов
  
  Step 2: Worker pool для сканирования
    - 4-8 воркеров для сканирования разных ветвей
    - Собирать результаты в единый список файлов
  
  Step 3: Балансировка нагрузки
    - Если одна директория намного больше, разделить еще
    - Или использовать work-stealing queue

ОЖИДАЕТСЯ:
  📊 На системах с медленным I/O: +10-15% производительность
  💾 Память: практически не изменится
  ⏱️ Пример: 65 сек → 55-58 сек

СЛОЖНОСТЬ:
  ⭐⭐⭐⭐ Высокая (требует правильной синхронизации, отладки на разных FS)

ВНИМАНИЕ:
  - Может замедлить на SSD (последовательное лучше)
  - Тестировать на HDD и SSD отдельно

════════════════════════════════════════════════════════════════════════════════

🔹 ОПЦИЯ 3: SIMD И AVX ОПТИМИЗАЦИЯ

ПРОБЛЕМА:
  - XXH3 хеширование может быть ускорено на CPU с SIMD поддержкой
  - Современные CPU имеют AVX2/AVX-512

РЕШЕНИЕ:
  - Использовать SIMD intrinsics для параллельного хеширования
    нескольких файлов одновременно

РЕАЛИЗАЦИЯ:
  Step 1: Проверить доступность SIMD
    - cpuid для получения CPU флагов
    - Fallback на стандартное XXH3 если нет поддержки
  
  Step 2: Batch хеширование
    - Вместо одного файла - один блок (SIMD width)
    - Параллелизм на уровне инструкций CPU
  
  Step 3: Zero-copy где возможно
    - Напрямую из mmap буфера в SIMD операции

ОЖИДАЕТСЯ:
  📊 На современных CPU (AVX2+): +5-8% производительность
  💾 Память: практически не изменится
  ⏱️ Пример: 65 сек → 60-62 сек

СЛОЖНОСТЬ:
  ⭐⭐⭐⭐⭐ Очень высокая (asm, cgo, cross-platform issues)

ВНИМАНИЕ:
  - Требует портирования на разные платформы
  - Возможны баги на старых CPU
  - Go SIMD support ограничен, может требовать cgo

════════════════════════════════════════════════════════════════════════════════

🔹 ОПЦИЯ 4: СПЕЦИАЛИЗИРОВАННОЕ ХЕШИРОВАНИЕ ПО ТИПАМ ФАЙЛОВ

ПРОБЛЕМА:
  - Разные типы файлов имеют разные характеристики:
    * Архивы (zip, rar) - заголовок часто уникален
    * Изображения (jpg, png) - мета-данные изменяются, контент нет
    * Текстовые файлы - последовательное хеширование неэффективно
  - Один размер не подходит всем

РЕШЕНИЕ:
  - Специализированная стратегия для каждого типа файла

РЕАЛИЗАЦИЯ:
  
  For Archives (zip, rar, 7z, etc):
    - Хешировать только content, пропускать компрессию-мету
    - Ожидается: +20-30% на архивах
  
  For Images (jpg, png, gif):
    - Пропускать EXIF и мета (часто отличаются, но контент одинаков)
    - Ожидается: +15-25% на изображениях
  
  For Text Files (txt, csv, log):
    - Использовать строковое сравнение вместо байтового
    - Пропускать BOM и кодировки
    - Ожидается: +10-15% на текстовых файлах
  
  For Duplicates:
    - Если fast_hash совпадает, можно пропустить full_hash
    - Использовать полиморфное сравнение

ОЖИДАЕТСЯ:
  📊 На смешанных наборах файлов: +5-10% производительность
  📊 На гомогенных наборах (только архивы): +20-30%
  💾 Память: +1-2 MB для таблицы типов
  ⏱️ Пример: 65 сек → 58-62 сек (зависит от типов)

СЛОЖНОСТЬ:
  ⭐⭐⭐ Средняя (но требует тестирования на разных типах)

ВОЗМОЖНЫЕ РИСКИ:
════════════════════════════════════════════════════════════════════════════════

⚠️ Кеширование может привести к stale данным:
   - Файл удален, но в кеше еще остался
   - Решение: проверка mtime + size перед использованием

⚠️ Параллельное сканирование может замедлить SSD системы:
   - I/O контроллер насыщен из-за большого параллелизма
   - Решение: автоматическое определение типа хранилища

⚠️ Специализированное хеширование усложнит код:
   - Больше тестирования нужно
   - Решение: хорошее модульное тестирование

⚠️ SIMD потребует кроссплатформенного тестирования:
   - Разные CPU имеют разные инструкции
   - Решение: fallback механизм
